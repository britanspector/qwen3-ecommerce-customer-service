# 电商客服场景大模型微调项目介绍

## 1. 项目概述

本项目是一个面向电商客服场景的大语言模型微调实践项目。项目基于 Qwen3-8B 模型，使用 HuggingFace Transformers、TRL 和 PEFT 框架，围绕电商客服中的真实业务问题，构建了从数据清洗、SFT-LoRA 监督微调、DPO 偏好优化到多维度效果评估的完整流程。

项目希望解决的问题是：通用大语言模型虽然具备较强的语言生成能力，但在垂直客服场景下，仍然容易出现回答泛化、业务处理不准确、缺少安抚语气、方案不可执行、甚至随意承诺退款赔偿等问题。因此，本项目通过领域数据微调和偏好优化，使模型更适合电商客服任务。

最终目标不是单纯让模型“说得更像客服”，而是让模型在以下几个方面得到提升：

1. 能够识别用户问题所属的客服场景；
2. 能够基于上下文生成符合业务逻辑的回复；
3. 能够使用礼貌、安抚、规范的客服表达；
4. 能够给出明确、可执行的处理建议；
5. 能够避免高风险承诺，例如随意退款、赔偿、补发。

---

## 2. 项目背景与问题分析

### 2.1 为什么需要做电商客服微调

在电商平台中，客服场景具有明显的垂直领域特征。用户的问题通常涉及订单、物流、商品参数、退换货、售后投诉等具体业务流程。相比一般聊天任务，客服任务对模型有更强的约束。

第一，客服回答必须准确。
如果用户询问商品规格、活动规则、退换货条件，模型不能随意编造信息，也不能答非所问。

第二，客服回答必须规范。
对于退款、赔偿、补发等问题，模型不能直接承诺，而应引导用户提供订单信息、核实情况或联系人工客服。

第三，客服回答必须有服务意识。
面对用户投诉或负面情绪时，模型需要先安抚，再解释原因，最后给出解决路径。

第四，客服回答需要可执行。
用户不只是需要一句“请耐心等待”，而是需要知道下一步应该怎么做，例如查看物流、申请售后、提供照片凭证、联系客服处理等。

因此，单纯使用原始通用模型并不能很好满足电商客服需求，需要通过领域数据进行微调。

### 2.2 本项目要解决的核心问题

本项目重点解决以下问题：

1. **模型回复不够场景化**原始模型能够生成通顺回答，但不一定符合电商客服的真实业务表达。
2. **回复结构不稳定**有时只有道歉，没有解决方案；有时直接给方案，缺少情绪安抚。
3. **风险话术控制不足**模型可能随意承诺退款、赔偿或补发，这在真实客服系统中存在风险。
4. **缺少面向客服任务的评估体系**
   传统 BLEU、loss 等指标不能完全反映客服回答质量，因此需要引入规则命中率、样本级胜率和模拟人工评分。

## 3. 数据来源与清洗

### 3.1 原始数据来源

本项目使用 GitHub `cooelf/DeepUtteranceAggregation` 中的电商客服对话数据作为基础数据来源。该数据包含用户与客服之间的多轮对话，并提供正负样本标签。

原始数据的特点是：

1. 来自真实或近似真实的电商客服对话；
2. 包含多轮上下文；
3. 包含正负样本标签；
4. 回复中存在大量口语化、碎片化和平台客服表达；
5. 数据质量参差不齐，需要进一步清洗。

### 3.2 原始数据不能直接用于 SFT 的原因

原始数据主要适合做候选回复匹配或对话排序任务，而不是直接用于大语言模型 SFT。主要原因包括：

1. 存在大量负样本；
2. 部分回复与上下文不匹配；
3. 部分回复过短，信息量不足；
4. 部分回复存在口语碎片和表达不规范问题；
5. 部分回答缺乏完整的客服处理流程；
6. 原始格式不是大模型训练常用的 instruction 或 messages 格式。

因此，项目首先进行了数据筛选与格式转换。

### 3.3 SFT 数据清洗流程

#### 3.3.1 删除负样本

原始数据中标签为 `0` 的样本表示候选回复与上下文不匹配。这类样本对于 SFT 阶段来说是不合适的，因为 SFT 需要模型学习“正确回答”，而不是学习错误回复。

因此，SFT 阶段只保留标签为 `1` 的正样本。

#### 3.3.2 删除重复问答

重复样本会导致模型过度学习某些固定表达，使模型输出更模板化。项目通过上下文和回答内容去重，减少重复数据对训练的影响。

#### 3.3.3 删除过短回答

一些原始客服回复过短，例如“好的哦”“没有呢”“可以的亲”等。这类回答虽然在真实客服场景中可能出现，但对于训练客服大模型帮助有限，容易让模型学到低信息密度回复。

因此，项目删除了回答长度小于 20 字的样本。

#### 3.3.4 删除前后矛盾或场景不匹配样本

部分样本虽然标签为正，但仍可能存在上下文不清、回答不完整、业务逻辑不一致等问题。项目进一步删除了明显不适合作为标准回复的样本。

#### 3.3.5 转换为 messages 格式

最终将清洗后的数据转换为 HuggingFace TRL 支持的 messages 格式，每条样本包含 system、user 和 assistant 三部分。

system prompt 用于约束模型角色：

```text
你是一名专业、礼貌、遵守平台规则的电商客服助手。回答需要先安抚用户，再给出清晰、可执行的处理建议，不能随意承诺退款、赔偿或补发。
```

user 部分包含任务指令、场景类别和对话上下文。

assistant 部分是期望模型学习的客服回复。

### 3.4 清洗后数据规模

经过清洗和筛选后，最终保留约 5800 条高质量 SFT 样本。

这些样本相比原始数据更加适合大模型训练，主要体现在：

1. 回复更完整；
2. 场景更清晰；
3. 上下文与答案更匹配；
4. 业务表达更规范；
5. 更适合学习客服回复结构。

## 4. SFT-LoRA 监督微调

### 4.1 为什么使用 LoRA

Qwen3-8B 模型参数规模较大，如果进行全参数微调，显存消耗和训练成本都较高。LoRA 是一种参数高效微调方法，它通过在模型部分线性层中插入低秩矩阵，只训练少量新增参数，从而降低训练成本。

使用 LoRA 的优势包括：

1. 显存占用更低；
2. 训练速度更快；
3. 便于保存和切换不同任务 adapter；
4. 适合个人项目和单卡 GPU 实验；
5. 能够较好地学习领域任务表达。

### 4.2 SFT 阶段的学习目标

SFT 阶段的目标是让模型学习电商客服场景下“应该如何回答”。具体包括：

1. 学习客服语气；
2. 学习电商常见业务表达；
3. 学习不同场景的问题处理方式；
4. 学习先安抚、再解释、最后给方案的回答结构；
5. 学习避免高风险承诺。

例如，面对用户投诉物流太慢时，模型不应只回答“请耐心等待”，而应生成类似结构：

```text
非常抱歉让您久等了，这边建议您先查看订单物流详情。如果物流长时间未更新，可以提供订单号，我们会帮您进一步核实快递状态，并根据实际情况协助处理。
```

这种回答同时包含：

1. 情绪安抚；
2. 问题解释；
3. 下一步动作；
4. 风险控制。

### 4.3 SFT 阶段预期效果

SFT 后，模型相比原始 Qwen3-8B 应该具有以下变化：

1. 更容易使用客服语气；
2. 更能围绕电商场景回答；
3. 回复更稳定；
4. 更少出现无关回答；
5. 能够初步遵守 system prompt 中的业务规则。

但是，SFT 本质上是模仿训练。如果训练数据中存在回答不够优的样本，模型也可能学到一些不理想模式。因此，SFT 后仍需要进一步评估和优化。

## 5. DPO 偏好优化

### 5.1 为什么 SFT 后还要做 DPO

SFT 可以让模型学会“像客服一样回答”，但不一定能保证模型总是生成“更优回答”。

例如，对于同一个用户问题，可能有两个回答：

回答 A：

```text
亲，不好意思，您再等等哦。
```

回答 B：

```text
非常抱歉给您带来不便。建议您先查看订单中的物流信息，如果超过预计发货时间仍未更新，可以提供订单号，我们会帮您核实仓库和快递状态，并根据平台规则协助处理。
```

两个回答都像客服，但 B 明显更好，因为它更完整、更可执行、更规范。

DPO 的作用就是通过 chosen / rejected 偏好数据，让模型更倾向生成 B 这类高质量回答，而不是 A 这类简单、模糊或不可执行的回答。

### 5.2 DPO 数据构造原则

本项目构造了约 2000 对 DPO 偏好数据，重点围绕以下问题改进模型：

1. 回答是否准确；
2. 是否安抚用户情绪；
3. 是否给出可执行方案；
4. 是否符合电商客服规范；
5. 是否避免高风险承诺；
6. 是否避免答非所问；
7. 是否避免空泛回复。

### 5.3 chosen 与 rejected 的判断标准

在构造 DPO 数据时，chosen 回复通常应满足：

1. 能准确理解用户诉求；
2. 能使用礼貌、专业的客服语气；
3. 能对用户情绪进行安抚；
4. 能给出具体处理步骤；
5. 不随意承诺退款、赔偿或补发；
6. 表达清晰，不含明显矛盾。

rejected 回复通常存在以下问题之一：

1. 回答过短；
2. 只安抚但不给方案；
3. 给出不符合规则的承诺；
4. 没有理解用户真实问题；
5. 回答与场景不匹配；
6. 表达生硬或不专业。

### 5.4 DPO 阶段优化目标

DPO 阶段的核心不是让模型记住更多客服文本，而是让模型学会区分“更好”和“更差”的客服回复。

最终希望模型在面对复杂问题时，更倾向输出：

```text
安抚用户 + 解释原因 + 明确步骤 + 风险控制
```

而不是输出：

```text
简单道歉 + 模糊等待 + 无明确方案
```

## 6. 评估体系设计

### 6.1 为什么不能只看 loss

在大模型微调项目中，loss 下降只能说明模型在训练数据上的拟合程度提高，但不能完全代表真实客服效果变好。

客服任务更关注：

1. 回答是否准确；
2. 是否符合业务规则；
3. 是否有安抚语气；
4. 是否能解决问题；
5. 是否存在风险话术。

因此，本项目没有只依赖训练 loss，而是构建了多维评估体系。

### 6.2 自动指标

#### 6.2.1 Char-BLEU

Char-BLEU 用于衡量模型输出与参考答案在字符级别上的重合程度。中文客服场景中，字符级 BLEU 比英文单词级 BLEU 更适合。

但是 BLEU 有明显局限：客服回答可能有多种正确表达，只要语义正确，不一定要和参考答案完全一致。因此 BLEU 只能作为辅助指标。

#### 6.2.2 BERTScore

BERTScore 用于衡量模型输出与参考答案之间的语义相似度。相比 BLEU，BERTScore 更关注语义层面的接近程度，因此更适合评估开放式客服回复。

本项目中，DPO 第二阶段模型的 BERTScore F1 从 0.7097 提升到 0.7260，相对提升 2.30%。

### 6.3 样本级胜率

样本级胜率用于比较两个模型在同一批测试问题上的输出优劣。

评估方式是：对每个测试样本，同时生成两个模型的回答，然后根据准确性、安抚能力、可执行性和业务规范性判断哪个回答更好。

实验中，第二阶段 DPO 模型在 65% 的评估样本上优于第一阶段模型。

相比单纯使用平均分，样本级胜率更直观，能够体现模型在实际案例中的相对优势。

### 6.4 规则命中率

规则命中率用于检查模型回答中是否包含关键客服行为。常见规则包括：

1. 是否包含安抚表达；
2. 是否包含处理建议；
3. 是否包含订单、物流、售后等业务动作；
4. 是否避免高风险承诺；
5. 是否没有出现明显违规承诺。

例如，一条较好的客服回复通常应包含：

```text
抱歉/不好意思/理解您的心情 + 建议您/可以先/请您提供 + 订单号/物流信息/售后申请/平台规则
```

规则命中率不能完全代替人工评估，但可以快速检查模型是否具备基本客服行为。

### 6.5 模拟人工评分

为了更贴近真实客服任务，本项目设计了模拟人工评分维度：

| 维度     | 说明                                       |
| -------- | ------------------------------------------ |
| 准确性   | 是否正确理解用户问题，并给出符合场景的回答 |
| 情绪安抚 | 是否对用户不满、疑问或投诉进行安抚         |
| 可执行性 | 是否给出具体、可操作的处理步骤             |
| 规范性   | 是否避免随意承诺退款、补发、赔偿           |
| 完整性   | 回答是否结构完整，是否遗漏关键处理环节     |

通过这些维度，可以比单纯自动指标更好地反映模型在客服场景下的真实表现。

## 7. 实验结果与分析

### 7.1 SFT 阶段效果

SFT 后，模型主要在以下方面有所提升：

1. 回复更接近客服话术；
2. 更容易围绕商品、物流、退换货等场景回答；
3. 回答语气更礼貌；
4. 输出结构更稳定；
5. 对 system prompt 中的规则约束有一定遵循能力。

例如，原始模型可能回答：

```text
这个问题你可以联系客服处理。
```

SFT 后模型更可能回答：

```text
非常抱歉给您带来不便。您可以先提供一下订单号和商品照片，我们这边帮您核实具体情况。如果符合平台售后规则，会协助您申请退换货处理。
```

这种回答更符合电商客服场景。

### 7.2 DPO 阶段效果

在 DPO 阶段，本项目重点优化回答准确性、情绪安抚和可执行方案生成能力。实验结果显示：

| 指标         | 第一阶段 DPO | 第二阶段 DPO |
| ------------ | -----------: | -----------: |
| BERTScore F1 |       0.7097 |       0.7260 |
| 相对提升     |            - |       +2.30% |
| 样本级胜率   |            - |          65% |

从结果可以看出，扩充偏好数据后，第二阶段 DPO 模型在语义匹配度和样本级对比中均优于第一阶段模型。

### 7.3 结果理解

BERTScore 的提升说明模型输出与参考答案在语义上更加接近。样本级胜率达到 65%，说明在多数测试样本中，第二阶段模型的回复更符合客服任务要求。

不过，这并不意味着模型已经完全达到工业级客服系统标准。当前项目仍属于微调流程验证和垂直任务适配实验，还需要进一步引入真实业务知识库、人工评估和上线安全策略。

## 8. 项目难点

### 8.1 数据质量控制

原始数据质量并不完全适合 SFT。项目中最大的难点之一是如何从较大的原始数据中筛选出适合训练的高质量客服回复。

如果数据清洗不严格，模型可能学到错误回复、短回复或不规范表达。

### 8.2 客服任务难以用单一指标评估

客服回答是开放式文本，不能只看 BLEU 或 loss。一个回答即使和参考答案不完全一样，也可能是合理的。因此需要结合语义指标、规则指标和人工维度评分。

### 8.3 DPO 偏好数据构造

DPO 的关键在于 chosen / rejected 是否真的体现质量差异。如果偏好数据质量不高，模型可能学不到正确偏好，甚至出现退化。

因此，本项目在构造 DPO 数据时重点关注回答准确性、安抚能力、方案可执行性和业务规范性。

### 8.4 避免过度承诺

客服模型不能直接承诺退款、赔偿或补发。这类风险话术在通用模型中可能出现，因此需要通过 system prompt、SFT 数据和 DPO 偏好数据共同约束。

---


## 9. 总结

本项目完成了一个较完整的电商客服大模型微调流程。项目不是简单调用模型接口，而是围绕真实垂直任务完成了数据处理、SFT-LoRA、DPO 偏好优化和多维评估。

从工程角度看，该项目体现了大模型微调的完整实践能力；从业务角度看，该项目围绕客服任务中的准确性、安抚性、规范性和可执行性进行了针对性优化；从简历角度看，该项目相比基础 MiniMind 学习项目更贴近实际业务场景，也更适合作为大模型算法或大模型应用岗位的重点项目。
