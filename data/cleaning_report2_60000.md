## 低质量样本过滤

本次清洗重点不是单纯保留“像客服”的话术，而是根据第一次 LoRA 失败经验，过滤容易导致模型学坏的样本。

主要去除类型如下：

* 过短回复：110770 条
* 重复 QA 样本：35969 条
* 纯商品编号 / 无意义问题：1251 条
* 只让用户核对地址的低质量回复：1602 条
* 明显风险话术：1133 条
* 闲聊 / 无关聊天：247 条
* 重复生成类回答：11 条

重点删除或降低权重的话术包括：

* “几颗坏果是正常现象哦”
* “可以再吃吃看”
* “退亲 5 元差价可以吗”
* “直接申请退款哦”
* “找快递索赔一下哦”
* “亲请核对一下收货地址哦”
* 大量重复的“亲亲 / 嗯嗯 / 好的呢”短回复

## 规则增强与样本扩充

由于原始客服数据中存在较多低质量口语化话术，本次额外加入了规则增强样本，用于纠正第一次 LoRA 出现的问题。

重点增强场景包括：

* 食品已拆封 + 非质量问题：一般不支持七天无理由
* 食品变质 / 有虫 / 发霉：不建议继续食用，保留照片或视频凭证，协助售后
* 服饰吊牌完整且未洗：可申请换码，运费按平台规则处理
* 物流 72 小时未更新：先核实物流，不直接判定丢件
* 已签收未收到：核实代收、驿站、快递柜、快递员，协助申诉
* 错发 / 漏发商品：提供照片、订单号、面单等凭证，核实后补发、退换或退款
* 商品参数不确定：不编造，以详情页、包装标识或官方参数为准

## 数据格式

目标数据支持 Hugging Face TRL 的 `messages` 训练格式，并额外加入 `category` 场景分类变量。

样本格式如下：

```json
{
  "category": "物流查询场景",
  "source": "raw_cleaned",
  "messages": [
    {
      "role": "system",
      "content": "你是一个专业、谨慎、负责的电商客服助手..."
    },
    {
      "role": "user",
      "content": "场景分类：物流查询场景\n用户问题：我的快递三天没有更新物流了，是不是丢件了？"
    },
    {
      "role": "assistant",
      "content": "您好，物流长时间未更新可能是中转延迟、扫描信息未同步或运输异常，不建议直接判断为丢件..."
    }
  ]
}
```

其中 `category` 同时出现在：

* 顶层字段：方便后续统计和评估
* user prompt：引导模型学习不同场景下的处理逻辑

## 总结

本次数据清洗的目标不是让模型更“亲亲化”，而是让模型在保持电商客服语气的同时，具备更好的规则判断、风险控制和场景化回复能力。
